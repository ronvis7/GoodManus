## 1. 项目概述

### 1.1 项目定位

GoodManus 是一个通用的、可完全私有部署的 AI Agent 系统。它采用 Planner-ReAct 多智能体协作架构，将用户的任务自动分解为多个子步骤，依次执行并在沙箱环境中完成各类操作——文件读写、Shell 命令、浏览器操控、搜索、代码编写等。同时通过 MCP（Model Context Protocol）和 A2A（Agent-to-Agent）协议支持扩展外部工具和远程 Agent。

## 2. 整体架构设计

### 2.1 Clean Architecture 分层设计

GoodManus 后端严格遵循 Clean Architecture（整洁架构）与 DDD（领域驱动设计）分层原则：

```
┌──────────────────────────────────────────┐
│              Interfaces 接口层             │  ← HTTP 请求入口
│  Routes · Schemas · Middleware · Errors   │     依赖方向 ↓
├──────────────────────────────────────────┤
│            Application 应用层              │  ← 业务编排
│  AgentService · SessionService · ...      │     无领域逻辑
├──────────────────────────────────────────┤
│              Domain 领域层                 │  ← 核心业务
│  Models · Agents · Flows · Tools · Protocols│   对外零依赖
├──────────────────────────────────────────┤
│          Infrastructure 基础设施层         │  ← 技术实现
│  PostgreSQL · Redis · LLM · Sandbox · COS │   实现领域协议
└──────────────────────────────────────────┘
```

**依赖规则：** 上层可以依赖下层，下层绝不能依赖上层。Domain 层是「纯内核」——不导入任何框架（FastAPI、SQLAlchemy），只定义 Python 原生类型和 Protocol 协议。

### 2.2 多智能体协作流程

GoodManus 的核心执行引擎是 Planner-ReAct 双 Agent 协作模式：

```
用户消息
    │
    ▼
┌─────────────────┐
│ Planner Agent    │  → 理解任务，拆解为 Plan（多个 Step）
│ (JSON Output)   │     输出: {title, goal, language, steps: [...]}
└────────┬────────┘
         │  Plan Created
         ▼
┌─────────────────┐
│ ReAct Agent     │  → 循环执行每个 Step
│ (Reason + Act)  │     Think → Tool Call → Observe → Think → ...
└────────┬────────┘
         │  Step Completed
         ▼
┌─────────────────┐
│ Planner Agent   │  → 根据执行结果更新 Plan
│ (Plan Update)   │     保留已完成步骤，替换未完成步骤
└────────┬────────┘
         │  Plan Updated
         ▼
┌─────────────────┐
│ ReAct Agent     │  → 执行下一个 Step
└────────┬────────┘
         │  ... 循环 ...
         ▼
┌─────────────────┐
│ ReAct Agent     │  → 所有 Step 完成后汇总总结
│ (Summarize)     │     输出最终回复 + 附件
└─────────────────┘
```

### 2.3 Flow 状态机

`PlannerReActFlow` 管理整个 Agent 任务的生命周期，状态转换如下：

```
IDLE → PLANNING → EXECUTING → UPDATING → EXECUTING → ... → SUMMARIZING → COMPLETED → IDLE
```

| 状态 | 含义 | 触发条件 |
|:-----|:-----|:---------|
| IDLE | 空闲 | 初始状态 / 任务完成 |
| PLANNING | 规划中 | 收到新消息，调用 PlannerAgent 创建 Plan |
| EXECUTING | 执行中 | Plan 生成完成，调用 ReActAgent 执行 Step |
| UPDATING | 更新中 | Step 执行完毕，调用 PlannerAgent 更新 Plan |
| SUMMARIZING | 汇总中 | 所有 Step 完成，调用 ReActAgent 生成总结 |
| COMPLETED | 完成 | 任务终结，重置为 IDLE |

## 5. Agent 多智能体系统

### 5.1 BaseAgent（基类）

位置：`api/app/domain/services/agents/base.py`

BaseAgent 是所有 Agent 的抽象基类，封装了与 LLM 交互、工具调用、记忆管理的通用逻辑。

**核心属性：**

| 属性 | 类型 | 说明 |
|:-----|:-----|:-----|
| `name` | `str` | Agent 唯一名称（planner / react） |
| `_system_prompt` | `str` | 系统预设 Prompt |
| `_format` | `Optional[str]` | LLM 响应格式（json_object） |
| `_tool_choice` | `Optional[str]` | 工具选择策略（none = 禁用工具） |
| `_retry_interval` | `float` | 重试间隔（默认 1 秒） |

**核心方法调用链：**

```
invoke(query)
  ├── _invoke_llm(messages)          ← 调用 LLM，含重试逻辑
  │     ├── _add_to_memory(messages)  ← 消息持久化
  │     └── llm.invoke(...)          ← OpenAI 兼容协议调用
  │
  └── for _ in range(max_iterations):
        ├── _get_tool(name)           ← 查找工具
        ├── yield ToolEvent(CALLING)  ← 工具即将调用
        ├── _invoke_tool(tool, ...)   ← 执行工具（含重试）
        ├── yield ToolEvent(CALLED)   ← 工具调用完成
        └── _invoke_llm(tool_results) ← 工具结果回传 LLM
```

**关键设计决策：**

1. **工具调用限制为每次 1 个**：防止 LLM 并行调用多个工具导致状态混乱（第 107 行 `message.get("tool_calls")[:1]`）
2. **空回复重试**：LLM 返回空 content 且无 tool_calls 时，自动注入"AI无响应内容，请继续"消息重试
3. **兼容 DeepSeek 思考模型**：保留 `reasoning_content` 字段（第 103-104 行）
4. **记忆自动压缩**：每完成一个 Step 调用 `compact_memory()`，防止上下文腐化

### 5.2 PlannerAgent（规划智能体）

位置：`api/app/domain/services/agents/planner.py`

```python
class PlannerAgent(BaseAgent):
    name = "planner"
    _system_prompt = SYSTEM_PROMPT + PLANNER_SYSTEM_PROMPT
    _format = "json_object"   # 强制 JSON 输出
    _tool_choice = "none"     # 禁用工具调用——规划阶段不需工具
```

**两个核心方法：**

#### `create_plan(message)` — 根据用户消息生成计划

1. 构造 `CREATE_PLAN_PROMPT`（含用户消息 + 附件）
2. 调用 `invoke()` → LLM 以 `json_object` 格式返回
3. 解析 JSON → `Plan` 模型（含 `title`, `goal`, `language`, `steps`, `message`）
4. 归一化 steps（处理 AI 可能返回字符串列表的边界情况）
5. 产出 `PlanEvent(CREATED)` + `TitleEvent` + `MessageEvent`

#### `update_plan(plan, step)` — 根据已完成的步骤更新计划

1. 将完整 Plan + 刚完成的 Step 序列化为 JSON
2. 调用 `invoke()` → LLM 返回更新后的 Plan
3. 合并策略：保留旧 Plan 中已完成的步骤，用新 Plan 替换未完成部分
4. 产出 `PlanEvent(UPDATED)`

### 5.3 ReActAgent（执行智能体）

位置：`api/app/domain/services/agents/react.py`

```python
class ReActAgent(BaseAgent):
    name = "react"
    _system_prompt = SYSTEM_PROMPT + REACT_SYSTEM_PROMPT
    _format = "json_object"  # 不使用 tool_choice 限制 → 允许工具调用
```

**三个核心方法：**

#### `execute_step(plan, step, message)` — 执行单个步骤

```
yield StepEvent(STARTED)
→ invoke(query) 进入 ReAct 循环
  ├── ToolEvent → 如果 function_name == "message_ask_user" → WaitEvent（中断等待用户输入）
  ├── MessageEvent → 步骤完成，解析结果 JSON
  └── ErrorEvent → 步骤失败
yield StepEvent(COMPLETED/FAILED)
```

#### `summarize()` — 汇总所有步骤结果

当所有子步骤执行完毕后，调用 LLM 生成最终总结回复（含附件路径）。

#### 特殊处理：`message_ask_user`

当 Agent 在工具调用中使用了 `message_ask_user`，系统不会继续执行而是产出 `WaitEvent`，等待用户通过前端输入新消息后继续。

### 5.4 PlannerReActFlow（编排流）

位置：`api/app/domain/services/flows/planner_react.py`

这是整个 Agent 系统的「大脑」，通过状态机编排 PlannerAgent 和 ReActAgent 的协作：

```python
class PlannerReActFlow(BaseFlow):
    def __init__(self, ...):
        self.planner = PlannerAgent(...)
        self.react = ReActAgent(...)
        self.status = FlowStatus.IDLE

    async def invoke(self, message: Message) -> AsyncGenerator[BaseEvent, None]:
        while True:
            if self.status == IDLE:
                self.status = PLANNING
            elif self.status == PLANNING:
                # PlannerAgent 创建 Plan
                async for event in self.planner.create_plan(message):
                    yield event
                self.status = EXECUTING
            elif self.status == EXECUTING:
                # ReActAgent 执行下一个 Step
                step = self.plan.get_next_step()
                async for event in self.react.execute_step(plan, step, message):
                    yield event
                await self.react.compact_memory()  # 压缩上下文
                self.status = UPDATING
            elif self.status == UPDATING:
                # PlannerAgent 更新 Plan
                async for event in self.planner.update_plan(plan, step):
                    yield event
                self.status = EXECUTING
            elif self.status == SUMMARIZING:
                # ReActAgent 总结
                async for event in self.react.summarize():
                    yield event
                self.status = COMPLETED
            elif self.status == COMPLETED:
                yield PlanEvent(COMPLETED)
                break
```

## 6. 工具系统

### 6.1 工具架构

所有工具继承 `BaseTool` 抽象类，使用 `@tool` 装饰器注册：

```python
class BaseTool(ABC):
    name: str                          # 工具箱名称（file / shell / browser / ...）

    def get_tools(self) -> List[Dict]   # 返回 OpenAI function calling 格式的工具声明
    def has_tool(self, name) -> bool    # 判断工具是否存在
    async def invoke(self, name, **kwargs) -> ToolResult  # 执行工具
```

`@tool` 装饰器自动将方法注册为可调用工具，并根据其签名和注解生成 OpenAI 兼容的 function schema。

### 6.3 FileTool 详解

位置：`api/app/domain/services/tools/tool.py`

FileTool 是对 Sandbox 文件操作的封装，所有操作实际在远程沙箱容器中执行。

| 工具名称 | 参数 | 说明 |

| `file_read` | filepath, start_line?, end_line?, sudo?, max_length | 读取文件（支持分页） |

### 6.4 MCP 客户端管理器

位置：`api/app/domain/services/tools/mcp.py`

`MCPClientManager` 负责管理所有 MCP 服务器的连接生命周期。

**支持的传输协议：**

| 传输方式 | 类 | 使用场景 |
|:---------|:---|:---------|
| `stdio` | `StdioServerParameters` | 本地命令行工具 |
| `sse` | `sse_client` | HTTP SSE 长连接 |
| `streamable_http` | `streamablehttp_client` | HTTP Streaming（项目主要使用） |

**连接流程：**

```
initialize()
  └── _connect_mcp_servers()
        └── for each server:
              _connect_{transport}_server(server_name, config)
                ├── enter_async_context(transport)
                ├── ClientSession(read, write)
                ├── session.initialize()
                └── _cache_mcp_server_tools()  ← list_tools() 缓存
```

**工具命名规则：**

MCP 提供的工具在实际暴露给 LLM 时，会被重命名为 `mcp_{server_name}_{tool_name}` 格式，以避免与其他工具箱冲突。例如 Jina AI 的 `search` → `mcp_jina-mcp-server_search`。

**invoke 流程：**

```
invoke("mcp_jina-mcp-server_search", {"query": "..."})
  ├── 解析 server_name = "jina-mcp-server"
  ├── 解析 tool_name = "search"
  ├── session.call_tool("search", {"query": "..."})
  └── 提取 result.content[].text
```

**资源清理：**

使用 `AsyncExitStack` 管理所有上下文（传输、会话），`cleanup()` 方法幂等安全，可多次调用。

### 6.5 A2A 客户端管理器

位置：`api/app/domain/services/tools/a2a.py`

`A2AClientManager` 通过 `httpx.AsyncClient` 与远程 A2A 服务通信。

**初始化流程：**

```
initialize()
  ├── 创建 httpx.AsyncClient(timeout=600)
  ├── 遍历 a2a_servers 配置
  └── GET {base_url}/.well-known/agent-card.json
        └── 缓存 agent_cards[id]
```

**调用远程 Agent：**

```python
async def invoke(agent_id: str, query: str) -> ToolResult:
    agent_card = self._agent_cards[agent_id]
    url = agent_card["url"]
    response = await self._httpx_client.post(url, json={
        "id": str(uuid.uuid4()),
        "jsonrpc": "2.0",
        "method": "message/send",
        "params": {
            "message": {
                "messageId": str(uuid.uuid4()),
                "role": "user",
                "parts": [{"kind": "text", "text": query}]
            }
        }
    })
```
